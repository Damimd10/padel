# Ticket Lifecycle

## Lifecycle

1. documented in product/architecture/testing docs
2. added to `docs/09-agile/product-backlog.md`
3. approved for execution
4. created or updated as a GitHub Issue
5. assigned lane labels, affected target metadata, milestone and project fields
6. selected into sprint
7. implemented by the workflow that matches its lane
8. reviewed and validated
9. closed in GitHub and marked done in docs

## Key rule

Tickets should not skip directly from documentation to implementation. GitHub Issues/Projects are the execution handoff layer between approved planning and coding.
